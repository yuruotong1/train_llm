# 我在 Windows 单卡上从零手搓了一个小语言模型：Pretrain → SFT → DPO 全流程实录

> 写在前面：这不是一篇「调 API 玩大模型」的文章，而是一篇「从一堆随机初始化的参数开始，亲手把它训成一个能对话的小模型」的实录。
>
> 主角是一个 **64M 参数**的小家伙（基于开源项目 MiniMind 的 minimind-3 结构）。我没有用 8 卡 A100，而是在一台 **Windows + 单张 32G 显卡**的台式机上，沿着 **预训练（Pretrain） → 监督微调（SFT） → 偏好对齐（DPO）** 这条主线，把它从「只会乱码」一路调到「能像模像样地聊天」。
>
> 这篇会分几个章节展开，先用比较大的篇幅把 **预训练**讲透，再讲 **SFT**、**DPO**，顺带聊聊我真正想做的事——把它改造成一个**检测领域的垂直小模型**。全程会贴上我实际跑的脚本、超参和踩过的坑。

---

## 〇、为什么是「从零」，又为什么是「这么小」

现在玩大模型，绝大多数人停在两个层面：要么直接调 GPT/DeepSeek/Qwen 的 API，要么顶多拿 LoRA 在别人的大模型上微调几条指令。这当然有用，但说实话，**这更像是教牛顿用智能手机——有趣，但你始终没碰到「物理本质」**。

模型内部到底发生了什么？为什么预训练完它就「有知识」了？SFT 到底改变了什么？DPO 那个一脸数学的损失函数，落到代码里其实就几行？这些问题，只有你真的从随机参数开始训一遍，才会有体感。

而「小」是关键前提。一个 64M 的模型：

- 单卡就能完整跑完整条链路，不需要租集群；
- 一轮预训练 1～2 小时，迭代实验的反馈极快；
- 所有核心算法（包括 DPO 的损失）都是纯 PyTorch 手写的，没有任何「十行代码黑箱」。

我的真实目标其实更具体：**我想要一个跑在自己机器上、懂「检测领域」术语和知识的垂直小模型**。但要做垂直模型，得先有一个通用底座，于是就有了这趟完整的从零训练之旅。

先把主角的「身材」交代清楚，后面才好聊训练。

## 一、先认识一下主角：一个 64M 的 Decoder-Only

这个模型的结构基本对齐了 `Qwen3` 的设计，是一个标准的 Transformer Decoder-Only，核心配置如下：

| 配置项 | 取值 | 说明 |
|--------|------|------|
| 参数量 | ≈ 64M | Dense 版本 |
| 隐藏维度 `hidden_size` | 768 | |
| 层数 `num_hidden_layers` | 8 | |
| 词表大小 | 6400 | 自定义 BPE 分词器 |
| 注意力头 | q_heads=8 / kv_heads=4 | GQA 分组查询注意力 |
| 位置编码 | RoPE（θ=1e6）+ YaRN 外推 | |
| 最大上下文 | 32768 | 训练时截断更短 |
| 归一化 / 激活 | Pre-Norm + RMSNorm / SwiGLU | |

几个值得说的点：

**1）为什么是「深 8 层、宽 768」这个身材？** 小模型的结构取舍其实有讲究。MobileLLM 那篇论文有个很有意思的结论：**参数量固定时，深度往往比宽度更重要**，「深而窄」比「宽而浅」更容易学到抽象概念。但「窄」也有下限——当 `hidden_size < 512`，词嵌入维度太窄的劣势会被放大，加再多层也补不回来。768 维 / 8 层，就是在训练速度、稳定性和效果之间的一个工程甜点。

**2）为什么词表只有 6400？** Qwen2 的词表是 15 万，Llama3 是 12.8 万，而我们只有 6400。对小模型来说，词表大小直接决定了 embedding 层和输出层的参数占比——词表越大，这两层就越「重」，挤占真正用来学知识的参数预算。6400 是个刻意压缩的选择，编解码效率虽然弱于大词表，但实测并没有明显的生僻字解码失败，对这个体量的模型反而更划算。

好，主角介绍完毕，开训。

## 二、第一章：预训练（Pretrain）——让模型先「读万卷书」

### 2.1 预训练到底在干一件什么事

一句话：**让模型学会高质量地「词语接龙」**。

听起来很简单，但魔力恰恰在这里。你给它海量文本——百科、新闻、书籍、对话语料——让它做一件唯一的事：**根据前面的词，预测下一个词**。输入「秦始皇」，它要学会接「是中国历史上第一位皇帝」；输入「为什么天空是」，它要学会接「蓝色的」。

这个阶段是**无监督**的。没有人逐条告诉它哪句对哪句错，它就在「猜下一个词→对答案→调整参数」这个循环里，自己把事实知识、语言模式、上下文的统计规律，一点点压进了那 64M 个参数里。

这是整条链路里最「重」也最关键的一步。**底座没打牢，后面 SFT、DPO 再怎么调都是空中楼阁。**

### 2.2 数据：长什么样，从哪来

预训练数据格式简单到「朴素」，就是一行一条文本：

```jsonl
{"text": "如何才能摆脱拖延症？治愈拖延症并不容易，但以下建议可能有所帮助。"}
{"text": "清晨的阳光透过窗帘洒进房间，桌上的书页被风轻轻翻动。"}
{"text": "Transformer 通过自注意力机制建模上下文关系，是现代大语言模型的重要基础结构。"}
```

我用的是 `pretrain_t2t_mini.jsonl`（约 1.2GB 的精简版，适合快速复现）。数据本身已经做过清洗、去重、长度控制和中英混合配比，来源是通用语料 + 对话整理语料 + 蒸馏补充语料。**「快速复现」用 mini 版，「认真训主线」才上完整的 10GB 版本。**

这里有个容易被忽略的细节——**token ≠ 字符**。我们这套分词器在中文上大约是 `1.5～1.7 字符/token`，纯英文大约 `4～5 字符/token`。所以设 `max_seq_len` 的时候，脑子里要换算：一段 500 字的中文，大概是 300 出头的 token。

### 2.3 训练目标与损失

预训练的损失就是最经典的**交叉熵**——预测的下一个 token 分布，和真实的下一个 token 之间的差距。代码里它长这样：

```python
res = model(input_ids, labels=labels)
loss = res.loss + res.aux_loss          # aux_loss 是 MoE 的负载均衡损失，Dense 版恒为 0
loss = loss / accumulation_steps        # 梯度累积
scaler.scale(loss).backward()
```

`res.loss` 就是 next-token-prediction 的交叉熵，也就是日志里看到的 `logits_loss`。`aux_loss` 只有 MoE 版本才非零（用来防止专家负载不均），Dense 版可以直接忽略。

### 2.4 几个关键超参，以及背后的「为什么」

预训练这步看着代码不多，但每个超参都有讲究：

**① 学习率：带「地板」的余弦退火**

学习率不是恒定的，而是按余弦曲线从峰值缓缓降到 10% 的「地板」。源码里就一行：

```python
def get_lr(current_step, total_steps, lr):
    return lr * (0.1 + 0.45 * (1 + math.cos(math.pi * current_step / total_steps)))
```

代入算一下：训练刚开始（step=0）系数是 `0.1 + 0.45×2 = 1.0`，正好是峰值学习率；训练到最后（step=total）系数是 `0.1 + 0.45×0 = 0.1`，降到峰值的 10%。**为什么留 10% 的地板而不是降到 0？** 因为完全降到 0 会让训练后期几乎不再学习，留一点底，让模型在收尾阶段仍有微调能力。预训练峰值学习率我用默认的 `5e-4`，对这个量级是个比较激进但稳妥的值。

**② 混合精度：bf16**

用 `bfloat16` 做混合精度，省显存、提速度。注意一个细节：`GradScaler`（梯度缩放）只在 `float16` 下才启用——因为 fp16 动态范围窄容易下溢，需要 scale；而 bf16 动态范围和 fp32 一样宽，不需要这个东西。

**③ 梯度累积 + 梯度裁剪**

`accumulation_steps` 让我用「小 batch 模拟大 batch」，把多个小 batch 的梯度攒起来再更新一次，等效放大 batch size 又不爆显存。`grad_clip=1.0` 则是把梯度范数裁到 1.0 以内，防止偶发的梯度爆炸把训练带飞——小模型训练前期尤其需要这个保险。

### 2.5 我的实际配置：一份 Windows 单卡的「安全版」脚本

官方的参考环境是 8 卡 3090 的 Ubuntu，而我是在 **Windows + 单张 32G 显卡**上跑的，踩了一些平台坑，最后固化成了这份脚本：

```powershell
# train_pretrain.ps1（Windows 安全版）
& "$root\venv\Scripts\python.exe" "$root\trainer\train_pretrain.py" `
    --data_path  "$root\dataset\pretrain_t2t_mini.jsonl" `
    --hidden_size 768 `
    --num_hidden_layers 8 `
    --batch_size 48 `       # batch16 只占 9G，加到 48 约占 21G，余量充足
    --num_workers 0 `       # ⚠️ Windows 上多进程 DataLoader 容易出问题，设 0 最稳
    --device cuda:0 `
    --dtype bfloat16 `
    --save_dir "$root\out"
```

两个 Windows 专属的经验：

- **`num_workers=0`**：Windows 的多进程 DataLoader 行为和 Linux 不一样，开多了容易出各种诡异问题（甚至蓝屏），单卡场景直接设 0 最省心。
- **显存别太抠**：batch16 才占 9G/32G，纯属浪费，直接拉到 48（约 21G）。真要是 `CUDA out of memory`，那也只是 Python 报个错，降回 32/16 就行，不会把系统搞崩。

还有一个非常隐蔽的坑：**Windows 上 `pyarrow` 和 `torch` 的 DLL 会冲突**，表现为一 import 就崩。解法是在所有 import 最前面先 `import datasets` 占个位（issue #771），这行注释我一直留着提醒自己别删。

### 2.6 断点续训：长训练的「保命」功能

预训练动辄几小时，万一中途断电、蓝屏、手贱关窗口怎么办？所以续训是刚需。我这套脚本所有阶段都支持 `--from_resume 1`，自动从 `checkpoints/` 里恢复**模型 + 优化器 + scaler + 训练进度**。

实现上有个细节很巧妙——它不是简单地从头跑，而是用一个 `SkipBatchSampler` **精确跳过已经训过的 batch**，从断点的下一个 step 接着练。甚至还支持**跨 GPU 数量恢复**（比如从 2 卡的快照恢复到 1 卡，它会自动换算 step）。对不稳定的家用环境来说，这个功能救过我好几次。

### 2.7 效果：预训练完，它「有知识」了吗

跑完预训练，先别急着 SFT，可以直接测一下底座（注意：此时它还**不会聊天**，只会续写）：

```text
💬: 为什么天空是蓝色的
🧠: 天空之所以看起来是蓝色的，主要是因为太阳光进入大气层后，短波长的蓝光更
    容易被空气分子散射，因此人眼从各个方向接收到的蓝光会更多。

💬: 解释什么是机器学习
🧠: 机器学习是人工智能的一个重要分支，它通过数据训练模型，使系统能够自动学习
    规律，并在分类、预测、推荐、自然语言处理等任务中持续改进效果。
```

看到没——**它已经「有知识」了**。瑞利散射、机器学习的定义，这些都不是谁教它的标准答案，而是它从海量文本里自己总结出来的。这一刻最有成就感：你亲手把一堆随机数，训成了一个「懂点东西」的东西。

### 2.8 我的支线：领域增量预训练（这才是我真正想要的）

通用底座有了，但我的目标是**检测领域**的垂直模型。直接拿领域数据从零训？数据太少（我手里就 30MB 左右），根本撑不起一个底座。正确姿势是**增量预训练（阶段二）**：

> 加载阶段一的通用底座 `pretrain_768.pth`，在「领域为主」的数据上**继续**预训练，把检测领域的知识、术语、文风注入进去，同时**尽量保留通用语言能力**。

脚本和通用预训练几乎一样，但有三个关键区别：

```powershell
# train_pretrain_domain.ps1
    --from_weight   pretrain          `  # ① 从通用底座接着练，不是从零(none)
    --learning_rate 1e-4              `  # ② 学习率调小，防「灾难性遗忘」
    --save_weight   pretrain_domain   `  # ③ 换个名字！否则会覆盖通用底座
```

这里面有两个关键认知：

- **学习率必须调小**（从 5e-4 降到 1e-4）。增量训练最大的敌人是**灾难性遗忘**——学新知识的时候把旧能力忘了。学习率小一点，相当于「轻踩油门」，慢慢渗透领域知识而不冲垮原有能力。
- **数据要「掺通用」**。领域数据不能 100% 纯用，否则模型会过拟合到那 30MB 上、丢掉通用语言能力。我的配方是：**领域数据过采样 N 遍 + 一部分通用数据，让领域占比控制在 15～25%**。这是用领域适配和防遗忘之间的平衡换来的。

数据不够，就**增量预训练 + 合成数据**两条腿走路——这是小数据做垂直模型唯一现实的路。

## 三、第二章：监督微调（SFT）——教它「好好说话」

### 3.1 SFT 不只是「调成会聊天」

很多人以为 SFT 就是把底座「格式化」成对话腔，其实远不止。

预训练出来的模型只会**续写**，你问它「你好」，它可能给你接一整段网页导航栏。SFT 要做的，是让它理解**「对话」这种交互形式**：谁是 user、谁是 assistant、什么时候该停。

但更重要的是——**像我用的这种 14GB 级别的 SFT 数据，本身已经不只是「格式对齐」，而更接近一种带 `mid-training` 性质的持续强化**。如果说预训练是「广泛读书」，SFT 就是「在高质量、有目标的数据上深加工」：一方面适应多轮对话、问答、工具调用的交互形式，另一方面继续把特定知识分布、任务模式和助手风格压进参数里。

### 3.2 数据格式：从 text 变成 conversations

SFT 数据从预训练的单条 `text`，变成了结构化的多轮对话：

```jsonl
{
  "conversations": [
    {"role": "user", "content": "你好"},
    {"role": "assistant", "content": "你好！"},
    {"role": "user", "content": "再见"},
    {"role": "assistant", "content": "再见！"}
  ]
}
```

训练时，`chat_template` 会把这些消息展开成带特殊标记的序列，模型逐渐理解 `system / user / assistant / tool` 这套角色结构。**关键点：损失只算在 `assistant` 的回复上**——我们要教它「怎么回答」，而不是「怎么提问」，所以 user 的内容不参与 loss 计算。

### 3.3 顺手白嫖：工具调用（Tool Call）

我用的 `sft_t2t_mini.jsonl` 里已经**混入了工具调用数据**，所以 SFT 完模型就自带基础的 Tool Call 能力，不用再单独训一轮。数据长这样：

```jsonl
{
  "conversations": [
    {"role": "system", "content": "# Tools ...", "tools": "[...]"},
    {"role": "user", "content": "帮我算一下 256 乘以 37 等于多少"},
    {"role": "assistant", "content": "", "tool_calls": "[{\"name\":\"calculate_math\",\"arguments\":{\"expression\":\"256 * 37\"}}]"},
    {"role": "tool", "content": "{\"result\":\"9472\"}"},
    {"role": "assistant", "content": "256 乘以 37 等于 9472。"}
  ]
}
```

`tools` 挂在 system 上，`tool_calls` 挂在 assistant 上，训练时模板自动展开成 `<tool_call>...</tool_call>` 和 `<tool_response>...</tool_response>`。这部分数据主要是用 `qwen3-4b` 合成的约 10w 条、覆盖约 10 个模拟工具（查时间、算数学、查天气等）——所以谈不上多强的泛化，但「会用工具」这个形态已经有了。

### 3.4 我的 SFT 配置

```powershell
# train_sft.ps1
    --from_weight   pretrain    `   # 在预训练权重基础上微调
    --save_weight   full_sft    `
    --epochs        2           `
    --learning_rate 1e-5        `   # 比预训练小得多
    --batch_size    48          `
    --dtype         bfloat16
```

注意 **SFT 的学习率 `1e-5` 比预训练的 `5e-4` 小了整整 50 倍**。逻辑很直白：预训练是「从零塑形」，可以大刀阔斧；SFT 是「在已成型的底座上精修」，动作必须轻柔，否则容易把预训练学到的知识冲掉。两个超参必须和预训练**完全一致**的是 `hidden_size` 和 `num_hidden_layers`——不然权重加载会直接错位。

### 3.5 效果：会聊天了，而且明显更「顺」

对比一下预训练 vs SFT 对同一个问题的回答：

```text
【预训练】解释什么是机器学习
🧠: 机器学习是人工智能的一个重要分支，它通过数据训练模型，使系统能够自动学习
    规律，并在分类、预测、推荐、自然语言处理等任务中持续改进效果。

【SFT 后】解释什么是机器学习
🧠: 机器学习是人工智能的核心技术之一，通过算法让计算机从数据中学习规律，并持续
    改进预测或决策效果，常见应用包括推荐系统、图像识别、语音识别和自然语言处理。

【SFT 后】推荐一些中国的美食
🧠: 例如北京烤鸭、兰州拉面、四川火锅、广东早茶、小笼包和麻婆豆腐等，这些美食
    分别代表了不同地区的风味特点，也很适合作为了解中国饮食文化的入门选择。
```

差别在于：SFT 后的回答**更像「在回答一个人的提问」**——结构更清楚、收尾更自然、有「推荐 + 点评」的助手腔。这就是 SFT 在干的事。

当然，这个体量的模型上限就摆在那。早期的 Zero 版本遇到没见过的分布（比如让它用英文讲美国历史）还会输出半通不通的内容：

```text
👶: Introduce the history of the United States, please.
🤖️: 您提到的"Introok's the believeations of theument."...
```

所以**别对 64M 抱不切实际的期待**——它是用来理解「训练这件事」的最佳教具，而不是用来对标 GPT 的产品。

## 四、第三章：偏好对齐（DPO）——教它「说人话、讨人喜欢」

### 4.1 为什么 SFT 之后还要 DPO

到 SFT 为止，模型的对话能力**完全建立在「单词接龙」上**——它学的是「优秀答案长什么样」，但**从来没见过「差答案长什么样」**。它不知道什么是好、什么是坏，缺少正反样例的对比激励。

DPO（Direct Preference Optimization，直接偏好优化）干的就是这件事：

> 就像让模型参加一次新培训，**拿优秀员工当正面例子、消极员工当反面例子**，告诉它「同样一个问题，这个回答（chosen）比那个回答（rejected）更好」，让它学会朝人类偏好的方向靠拢。

数据格式就是一对一对的「偏好对」：

```json
{
  "chosen":   [{"role":"user","content":"Q"}, {"role":"assistant","content":"好答案"}],
  "rejected": [{"role":"user","content":"Q"}, {"role":"assistant","content":"差答案"}]
}
```

我用的 `dpo.jsonl` 里有 **17166 条**这样的偏好对（抽样自 DPO-En-Zh-20k）。

### 4.2 DPO 的损失：看着吓人，写出来就几行

DPO 的损失函数公式长这样：

$$\mathcal{L}_{DPO} = -\mathbb{E}\left[\log \sigma\left(\beta \left[\log \frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \log \frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right]\right)\right]$$

一堆希腊字母，但直觉很简单：**让「策略模型」给 chosen 的概率相对 rejected 拉高，同时用一个「冻结的参考模型」当锚点，不让它跑太偏。** 落到 PyTorch 里就是这几行（纯手写，没用任何 RL 框架）：

```python
def dpo_loss(ref_log_probs, policy_log_probs, mask, beta):
    ref_log_probs    = (ref_log_probs * mask).sum(dim=1)
    policy_log_probs = (policy_log_probs * mask).sum(dim=1)

    # 一个 batch 里前一半是 chosen，后一半是 rejected
    n = ref_log_probs.shape[0] // 2
    chosen_ref,  reject_ref  = ref_log_probs[:n],    ref_log_probs[n:]
    chosen_pi,   reject_pi   = policy_log_probs[:n], policy_log_probs[n:]

    pi_logratios  = chosen_pi  - reject_pi      # 策略模型：chosen 比 rejected 好多少
    ref_logratios = chosen_ref - reject_ref     # 参考模型：chosen 比 rejected 好多少
    logits = pi_logratios - ref_logratios       # 两者之差，就是 DPO 要优化的核心量
    return -F.logsigmoid(beta * logits).mean()
```

DPO 的妙处在于：它从带 KL 约束的 PPO 目标里**解析地推导**出了对偏好对的训练目标，**完全不需要单独训练 Reward / Value 模型**。整个过程只跑两个模型——可训练的 `actor`（策略模型）和冻结的 `ref`（参考模型），显存占用低、收敛稳定、实现简单。

### 4.3 我的 DPO 配置：注意那个「离谱」的学习率

```powershell
# train_dpo.ps1
    --from_weight   full_sft   `   # 同时作为「策略模型」和「冻结参考模型」的起点
    --save_weight   dpo        `
    --epochs        1          `
    --learning_rate 4e-8       `   # ← 学习率小到离谱！
    --beta          0.15       `   # 控制偏离参考模型的程度
    --batch_size    8              # 显存吃紧：同时常驻 2 个模型 + chosen/rejected 拼接
```

最反直觉的就是这个学习率 **`4e-8`**——比 SFT 的 `1e-5` 又小了 **250 倍**。为什么这么夸张？因为 **DPO 极其敏感，学习率稍微大一点就会「灾难性遗忘」**，把 SFT 辛辛苦苦练出来的对话能力直接冲垮。在这个阶段，我们要的不是「大改」，而是「极其轻微地把天平往人类偏好那边压一点点」。

另外两个工程点：

- **`from_weight` 同时初始化了两个模型**：策略模型（要训练的）和参考模型（冻结、`requires_grad=False`），都从 `full_sft` 出发。参考模型全程不更新，只用来提供「锚点」概率。
- **显存比 SFT 更紧**：因为要同时常驻两个模型，还要把 chosen / rejected 拼在一起前向，所以 batch 给到 8 就行，别贪大。

### 4.4 关于 DPO 效果的「诚实话」

这里我想说点实在的：**在 64M 这个量级、用这种规模的偏好数据，DPO 带来的提升是「微妙」的，不要期待质变。**

DPO 的本质是**偏好和安全层面的价值对齐**——让回答更得体、更稳、更贴合人类喜好，降低「让人不满意的答案」的出现概率。但它**对「能不能把题做对」这种硬核智力提升帮助有限**（这也取决于数据，大规模收集正反样例并做人类评估本身就极难）。

所以我的态度是：DPO 在这条链路里更多是**「完整体验一遍 RLHF 流程、理解偏好对齐原理」**的价值，而不是指望它让小模型脱胎换骨。想真正提升「做对题」的能力，得往后看——RLAIF / GRPO / 可验证奖励（RLVR）那条线。

## 五、后面还有什么（一笔带过）

主线 Pretrain → SFT → DPO 走完，模型已经是个能聊天、会调工具的「成品」了。再往后，这套链路还铺了很多可选支线，简单提一下：

- **LoRA**：参数高效微调，只训练低秩增量分支，特别适合「在保留通用能力的前提下，低成本适配垂直领域」——我后续给检测模型做轻量适配大概率会用到它。
- **知识蒸馏**：黑盒蒸馏（学强模型的答案/风格）+ 白盒蒸馏（连 token 分布偏好一起学）。
- **RLAIF（PPO / GRPO / CISPO）**：用 AI 反馈或可验证奖励（答案对不对、代码过不过测试）做强化学习，这才是提升「智力」的主战场。
- **Agentic RL**：多轮工具使用场景下的强化学习，奖励来自多步交互后的延迟结算。
- **自适应思考**：把「要不要显式输出思考过程」下沉到 `chat_template`，用一个 `open_thinking` 开关动态切换，不再单独训一个「思考模型」。

这些每一个都能单独写一篇，这里先按下不表。

## 六、训练开销 & 一点总结

最后给个成本概念。以单卡 3090 的官方经验估算（我自己在 Windows 单卡上量级类似）：

| 阶段 | 64M Dense 单卡耗时 | 约合成本 |
|------|-------------------|----------|
| 预训练（mini） | ≈ 1.21 h | ≈ 1.6 元 |
| SFT（mini） | ≈ 1.10 h | ≈ 1.4 元 |
| Tool Call | ≈ 0.9 h | ≈ 1.2 元 |
| RLAIF | ≈ 1.1 h | ≈ 1.4 元 |

也就是说，**「预训练 + SFT」从零训出一个能聊天的对话模型，单卡 2 小时出头、3 块钱搞定。** 这个门槛低到任何一个有张游戏显卡的人都能复现。

回头看这趟从零训练，我最大的感受是：

1. **预训练决定上限，后训练决定下限。** 知识是预训练灌进去的，SFT/DPO 只是把这些知识「以更好的方式表达出来」。底座不行，后面调不动。
2. **每一步的「学习率」都是性格。** 5e-4（预训练，大刀阔斧）→ 1e-4（领域增量，轻踩油门）→ 1e-5（SFT，精修）→ 4e-8（DPO，如履薄冰）。学习率从大到小的变化，本身就是「从塑形到对齐」的过程缩影。
3. **算法没那么神秘。** DPO 的损失写出来就几行，所谓 RLHF 也不过是「正反例对比」。真正难的、占 90% 时间的，是数据清洗、防遗忘配比、平台踩坑（尤其 Windows）、续训保命这些「脏活」。

对我来说，通用底座只是起点，下一步是沿着「**领域增量预训练 + 合成数据 + LoRA 适配**」这条线，把它真正打磨成一个懂检测领域的垂直小模型。那会是另一篇文章的故事了。
